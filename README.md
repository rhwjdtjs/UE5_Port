Unreal Engine 5 Listen Server Multiplayer Game  
  
UE5 기반 Listen Server 구조의 멀티플레이 TPS 게임입니다.  
서버 권위(Server Authoritative) 모델을 기반으로 RPC, Replication, Steam Online Subsystem을 직접 구현하는 것을 목표로 개발했습니다.  
  
본 프로젝트는 AWS Dedicated Server 없이도 실제 멀티플레이 흐름(세션 생성 → 로비 → 매치 → 전투 → 종료)을 완성하는 데 초점을 맞췄으며 향후 Dedicated Server(AWS / EOS / GameLift)로 무리 없이 확장 가능한 구조로 설계되어 있습니다.  
  
핵심 특징  
  
* Listen Server 기반 멀티플레이  

* Server Authoritative 구조  

* Steam Online Subsystem(OSS) 세션 / 매칭  

* RPC & Replication 설계 원칙 명확화

* UI / 네트워크 / 게임플로우 분리 설계  

* 실전 멀티플레이 디버깅 경험 반영  
  
네트워크 아키텍처 개요  
서버 권위 모델 (Server Authoritative)  

클라이언트는 입력만 전달  

모든 판정(대미지, 장착, 사망, 리스폰)은 서버에서만 확정  
  
결과는 Replication 또는 RPC를 통해 클라이언트에 전파  
  
Client Input  
   ↓  
Server RPC (검증)  
   ↓  
Server Logic  
   ↓  
Replication / Client RPC  
   ↓  
Client HUD & Effects  
  
RPC / Replication 설계 원칙  
  
지속 상태 (체력, 점수, 무기 등)	Replication + OnRep	늦은 입장에도 상태 보장  
개인 UI / 사운드	Client RPC	대상 필터링 용이  
짧은 연출 (이펙트, 섬광)	NetMulticast (Unreliable)	즉각 동기화  
고빈도 이벤트	로컬 처리 + 서버 검증	대역폭 절약  
  
Multicast는 “연출용”으로만 사용  
상태(State)는 반드시 Replication으로 관리  
개인화 요소는 PlayerController Client RPC 사용  
Steam Online Subsystem (OSS)  
세션 생성 / 검색 / 참가를 빠르게 구현  
NAT / 방화벽 문제 완화 (Steam 네트워크 릴레이)  
별도 서버 인프라 없이 멀티플레이 테스트 가능  
  
UGameInstanceSubsystem 기반 MultiplayerSessionsSubsystem  
  
UI는 Subsystem을 직접 접근하지 않고 델리게이트 구독 방식  

UMenu (UI)  
  ↓  
MultiplayerSessionsSubsystem  
  ↓  
Steam OSS  
  
세션 흐름  
  
Host가 세션 생성  
  ↓  
ServerTravel("Lobby?listen")  
  ↓  
Host = Server + Player  
   
클라이언트 세션 검색  
  ↓  
MatchType 필터링  
  ↓  
ClientTravel(Address)  

UI / 네트워크 분리 설계  
  
네트워크 로직: PlayerController / GameMode / Subsystem  
  
UI 로직: HUD / Widget  
  
연결은 델리게이트 + RPC로만 수행  
  
PlayerController 역할  
  
PlayerController는 플레이어 UI의 중심이다.  
HUD 수치 갱신 (체력, 실드, 탄약, 점수)
채팅 브로드캐스트 중계  
킬피드 / 알림 / 스코어보드 표시  
매치 상태(MatchState) 기반 UI 전환  
서버 시간 동기화 (RTT 기반)  
HUD 초기화 안정성 (PollInit 패턴)  
UE 특성상 HUD / Pawn / Replication 순서는 비동기입니다.
이를 해결하기 위해 PollInit() 패턴을 사용했습니다.  
PollInit 개념: HUD가 준비될 때까지 주기적으로 확인하고 값은 미리 캐싱
HUD 생성 시 한 번에 밀어넣고 초기화 완료 후 타이머 종료  
시간 동기화 (Server Time Sync): RTT(Round Trip Time) / 2 방식  
서버 기준 시간을 클라이언트에서 추정하여 모든 클라이언트 타이머가 같은 속도로 감소  
Client Time + Delta = Server Time → 매치 타이머 / 쿨다운이 모든 클라에서 일관되게 동작  
  
기술 요약 (TL;DR)  
* UE5 Listen Server 멀티플레이 구현  
* Server Authoritative + RPC / Replication 명확한 분리  
* Steam OSS 기반 세션 / 로비 / 매칭  
* UI / 네트워크 / 게임플로우 계층 분리  
* 실제 멀티플레이 디버깅 경험 반영
